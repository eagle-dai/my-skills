# 如何安全改进这个 skill（元规则）

改这个 skill 本身时读——不是转换文档时读。目标：防两个反复发生的病根。

| 病根 | 症状 | 真实案例 |
|------|------|---------|
| **泛化不够** | 从单个样例提炼规则，规则贴死那个样例的表面特征，换一类输入就错 | 缺陷 7：从"英文紧贴 `**`"样例提出 `\S` 检测，泛化到中文时把 CJK 标点也当违规 |
| **改坏旧样例** | 修 A 缺陷时改了共享规则/正则，没验证 B 场景（旧的、已过的）还对不对 | 修缺陷 6（加空格）时没回归"句末加粗"，缺陷 7 才暴露过度修复 |

两病根同源：**skill 改进是无 CI 的回归修改**。规则是纯文本，改了不会报错，坏了要下次转换才发现。所以靠两道人工闸：改前答泛化清单，改后跑回归用例表。

---

## 闸 1：泛化检查清单（改任何规则/正则前必答）

新增或修改一条规则前，逐条自问。任一"否"→ 规则还没准备好，别写进 skill。

1. **样例覆盖几类？** 手里的触发样例是 1 个还是 ≥3 个不同类别（中文/英文/数字/标点/符号/全角/半角…）？只有 1 类就**别急着定正则**——先构造反例试探边界。
2. **规则依据的是「机制」还是「表面特征」？** 写下规则为什么成立的**机制**（如"CommonMark right-flanking：闭合定界符后是 punctuation 仍有效"）。只会说"因为这个样例这样"=贴死表面特征，会误伤。机制说不清→规则不可信。
3. **构造了「不该命中」的反例吗？** 每条检测规则至少配 2 个**期望不命中**的样例。检测正则最容易犯的错是**过宽**（假阳性），假阳性会诱导下游 sub agent 做**过度修复**、反引入新 bug。
4. **字符类精确到位了吗？** 涉及字符边界的规则（`$`/`*`/`_`/`#`/NBSP…），必须精确到"紧贴的是**哪一类**字符"（字母 vs 数字 vs CJK 汉字 vs CJK 标点 vs ASCII 标点 vs 空白）。用 `\S`/`.`/`\w` 这类**宽类**几乎总是过宽——先怀疑它。
5. **平台差异标了吗？** 目标平台（GitHub 严格 / VS Code 宽容 / 本地 KaTeX）行为不同的，规则里注明"哪个平台触发"。别用单一渲染器结论当通用结论（这是缺陷 3 的根因）。

## 闸 2：回归用例表（改后必须全通过，禁止删已有行）

**核心机制：每条边界/清理规则都在下表登记「输入→期望命中/不命中」。改任何检测正则后，必须让全表通过再落地。**新增规则→加行（含至少 1 正例 + 2 反例）；改规则→跑全表；**永远不删已有行**（删行=丢回归保护，除非该场景被证伪并写明原因）。

跑法：把下表喂给一段 Python，逐行 `re.search`，`hit != expect` 即 FAIL。几秒跑完。

### 强调定界符边界（缺陷 6/7，已被烧两次，重点保护）

当前正则：`(\*\*[^*\n]+?\*\*)([^\s。，、；：！？）】」』.,;:!?)])`

| 输入 | 期望命中 | 类别 | 理由 |
|------|:---:|------|------|
| `**标签：**https://` | ✅ | 英文字母紧贴 | right-flanking 失败，真违规 |
| `**概念辨析：**为什么` | ✅ | CJK 汉字紧贴 | 真违规 |
| `**结论**。` | ❌ | CJK 句号紧贴 | 标点是有效 right-flanking，GitHub 渲染正常 |
| `**结论**，后面` | ❌ | CJK 逗号紧贴 | 同上 |
| `**记录**：写清` | ❌ | 全角冒号紧贴 | 同上 |
| `**读法**。只看` | ❌ | 句号+汉字 | 句号先紧贴→合法 |
| `**结论** 。` | ❌ | 已插空格 | 不该再命中（此形态另用反向正则清理） |

反向清理正则（清缺陷 7 过度修复留下的错误空格）：`\*\*[^*\n]+?\*\* [。，、；：！？）]` —— 命中即删空格。

### 行内公式 `$` 边界（缺陷 3，GitHub 平台）

当前正则：`([一-鿿　-〿＀-￯])\$(?!\$)` / `(?<!\$)\$([一-鿿　-〿＀-￯])`

| 输入 | 期望命中 | 类别 | 理由 |
|------|:---:|------|------|
| `收益率$r_t$表示` | ✅ | CJK 紧贴 $ 两侧 | GitHub 不渲染 |
| `见 $r_t$ 公式` | ❌ | ASCII 空格隔开 | 正常 |
| `（$w_t$）` | ✅ | 全角括号紧贴 | GitHub 不渲染 |

### 数学块内裸 `*`（缺陷 5，GitHub 平台）

检测：数学段内 `(?<!\\)\*` 命中 → 阻断，修 `*`→`\ast`

| 输入 | 期望命中 | 理由 |
|------|:---:|------|
| `$SR^{*}$` | ✅ | 裸 `*` 被当 emphasis，MathJax 报错 |
| `$a \ast b$` | ❌ | 已转义 |

> 新增规则请在此文件按同样格式加小节 + 用例。用例是这个 skill 的**回归测试套件**，价值随行数增长。

## 改进落地流程

1. 读病根表 + 相关 memory（`reference-html-to-md-skill-gaps`）确认不是重复踩坑。
2. 过**闸 1 泛化清单**——构造 ≥3 类样例 + ≥2 反例，写清机制。
3. 改规则/正则。
4. 过**闸 2 回归用例表**——跑全表 Python，全绿才落地。改了共享正则尤其要跑全表（别只测新场景）。
5. 同步更新受影响的所有文件（规则常散落 SKILL.md 模板 / conversion-rules / blocking-rules / checklist **多处**——`grep` 全仓找同一正则，一处不漏，否则 sub agent 读到旧的那份就白改）。
6. 更新 `reference-html-to-md-skill-gaps` memory 记录本次缺陷 + 根因。

**教训沉淀**：缺陷 3/5/6/7 同族=markdown 特殊符与内容边界冲突。但**方向会反**——6 是漏检（规则太窄），7 是假阳性（规则太宽/`\S`）。边界规则的对错永远取决于"紧贴的是哪类字符"，字符类不精确，修复动作自身就是下一个 bug 源。
