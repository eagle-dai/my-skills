# 如何安全改进 skill（通用元规则）

改**任何自建 skill** 本身时读——不是执行 skill 任务时读。纯文本规则不会像编译代码那样天然暴露错误；本仓库已经用 GitHub Actions 执行部分可机器判定的合同与回归测试，但语义判断、渲染质量和截图对比仍需要人工闸。各 skill 的专属回归用例放在其目录的 `self-improvement.md`。

## 两个反复发生的病根

| 病根 | 症状 |
|------|------|
| **泛化不够** | 从单个样例提炼规则，规则贴死表面特征，换一类输入就错 |
| **改坏旧样例** | 修 A 缺陷时改了共享规则/正则，没验证 B 场景仍然正确 |

自动化测试负责确定性合同；人工检查负责难以机器判定的语义和视觉质量。两者都不能省略。

---

## 闸 1：泛化检查清单

任一“否”都说明规则还没准备好。

1. **样例覆盖几类？** 至少构造多个类别，包括中文、英文、数字、标点、全角/半角或大小写差异。
2. **依据机制还是表面特征？** 写清标准、渲染器或 DOM 机制；只会描述当前样例不够。
3. **有反例吗？** 每条检测规则至少补 2 个期望不命中的样例，防止假阳性和过度修复。
4. **字符类足够精确吗？** 涉及边界时区分字母、数字、CJK、标点和空白；优先怀疑 `\S`、`.`、`\w` 等宽类。
5. **平台差异标明了吗？** GitHub、VS Code、KaTeX、MathJax 等行为不同时，必须注明适用平台。

---

## 闸 2：自动化与人工回归

每条可机器判定的规则应落到 Python 合同或测试中；专属输入→输出案例仍登记在该 skill 的 `self-improvement.md`。

- 新增规则：增加正例和至少 2 个反例。
- 修改规则：运行完整测试集，而不是只跑新增案例。
- 已有案例原则上不删除；若场景被证伪，记录原因后再调整。
- 文档中的 selector、正则、默认行为若有可执行实现，以实现和测试为权威来源。

执行：

```bash
python -m pip install -r requirements.txt
python -m unittest discover -s tests -p 'test_*.py' -v
```

GitHub Actions 会在 pull request 和 `main` 推送时执行相同测试。仍需人工验证的内容包括：列表 marker 语义、公式结构重建、图片内容完整性、渲染截图和站点专属 adapter。

---

## 改进落地流程

1. 读病根表和相关专属回归表，确认不是重复踩坑。
2. 通过泛化检查：构造多类正例与反例，写清机制。
3. 把确定性规则落到可执行模块；同步修改自然语言说明。
4. 运行完整测试集并检查 GitHub Actions。
5. 对语义和视觉规则执行计数、渲染与截图验证。
6. 全仓搜索同一 selector、正则和默认行为，清除冲突副本。
7. 在 PR 描述中写明根因、风险、验证方式和未覆盖范围。

**通用教训：** 边界和特殊符规则的正确性取决于具体字符类别。字符类不精确，修复动作本身就会成为下一个 bug 源。
