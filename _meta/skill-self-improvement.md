# 如何安全改进 skill（通用元规则）

改**任何自建 skill** 本身时读——不是执行 skill 任务时读。skill 是纯文本规则，改了不报错、坏了要下次用时才发现——**本质是无 CI 的回归修改**。这份文件给通用的两道人工闸；各 skill 的**专属回归用例表**放在该 skill 目录自己的 `self-improvement.md` 里。

## 两个反复发生的病根

| 病根 | 症状 |
|------|------|
| **泛化不够** | 从单个样例提炼规则，规则贴死那个样例的表面特征，换一类输入就错 |
| **改坏旧样例** | 修 A 缺陷时改了共享规则/正则，没验证 B 场景（旧的、已过的）还对不对 |

两者同源：无自动化测试托底。所以靠两道人工闸——改前答泛化清单，改后跑回归用例表。

---

## 闸 1：泛化检查清单（改任何规则/正则前必答）

任一"否"→ 规则还没准备好，别写进 skill。

1. **样例覆盖几类？** 触发样例是 1 个还是 ≥3 个不同类别（中文/英文/数字/标点/符号/全角/半角/大小写…）？只有 1 类就**别急着定正则**——先构造反例试探边界。
2. **依据「机制」还是「表面特征」？** 写下规则为什么成立的**机制**（引哪条标准、哪个渲染器的哪条行为）。只会说"因为这个样例这样"=贴死表面特征，会误伤。机制说不清→规则不可信。
3. **构造了「不该命中」的反例吗？** 每条检测规则至少配 2 个**期望不命中**的样例。检测正则最常犯的错是**过宽**（假阳性），假阳性会诱导下游做**过度修复**、反引入新 bug。
4. **字符类精确到位了吗？** 涉及字符边界的规则，必须精确到"紧贴/涉及的是**哪一类**字符"（字母 vs 数字 vs CJK 汉字 vs CJK 标点 vs ASCII 标点 vs 空白 vs 特定 Unicode 段）。用 `\S`/`.`/`\w` 这类**宽类**几乎总是过宽——先怀疑它。
5. **平台差异标了吗？** 目标平台行为不同的（如 GitHub 严格 / VS Code 宽容 / 本地渲染器 / KaTeX vs MathJax），规则里注明"哪个平台触发"。别用单一渲染器结论当通用结论。

---

## 闸 2：回归用例表（各 skill 自己维护，改后必须全通过）

**机制：每条可机器判定的规则（正则检测/文本替换）都在**该 skill 的** `self-improvement.md` 登记「输入→期望命中/不命中」或「输入→期望输出」。改任何该类规则后，必须让全表通过再落地。**

- 新增规则 → 加行（含至少 1 正例 + 2 反例）。
- 改规则 → 跑全表（不只测新场景，尤其改了共享正则）。
- **永远不删已有行**（删行=丢回归保护，除非该场景被证伪并写明原因）。

跑法：把用例表喂给一段临时 Python，逐行 `re.search`（检测类）或 `re.sub` 比对（替换类），`结果 != 期望` 即 FAIL。几秒跑完。

**局限（老实说）**：
- 无 hook/CI 强制，靠改 skill 的人记得跑 + 记得读这份文件。
- 只覆盖**可机器判定**的规则（正则/替换）。语义规则（列表 marker 判对没、公式重建对没、图片去水印没擦到内容）测不了正则，只能靠 skill 自身的实际执行验证（截图对比、计数对比）兜底。
- 用例是手填的，不是从历史执行自动抽的。真正的端到端回归需要标注好的样例库，成本高，暂未建。

---

## 改进落地流程

1. 读病根表 + 相关 memory，确认不是重复踩坑。
2. 过**闸 1 泛化清单**——构造 ≥3 类样例 + ≥2 反例，写清机制。
3. 改规则/正则。
4. 过**闸 2 回归用例表**（该 skill 自己的 `self-improvement.md`）——跑全表 Python，全绿才落地。
5. **同步更新受影响的所有文件**——同一规则常散落 SKILL.md 模板 / 各 rules 文档 / checklist **多处**，`grep` 全仓找同一正则/字符串，一处不漏，否则下游读到旧那份就白改。
6. 更新对应 gaps memory 记录本次缺陷 + 根因 + 方向（漏检=规则太窄 / 假阳性=规则太宽）。

**通用教训**：边界/特殊符规则的对错永远取决于"涉及的是哪类字符"。字符类不精确，**修复动作自身就是下一个 bug 源**——同一族缺陷方向会反（漏检 ↔ 假阳性），别把上次的修法当这次的答案。
